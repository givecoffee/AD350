Course summary

AD350 covers the foundations of database technology from a developer's perspective. The relational model is the anchor — schema design, normalization, SQL, transactions, indexing — and the term extends into the wider landscape of document stores, key-value caches, and the tradeoffs that drove their emergence.

The work happens in two parallel tracks. One is conceptual: readings, discussion posts, and interview-style questions to sharpen vocabulary and intuition. The other is practical: a term-long project that begins with an entity model on paper and ends with a working application backed by a real database, deployed and queryable.

Working principle

The schema is a hypothesis about how the world holds together. Everything downstream — queries, indexes, application code, the user experience itself — pays the price of getting it wrong, and inherits the leverage of getting it right.

Week 03Jan 28

Foreign keys aren't free

Spent an afternoon chasing an insert error that turned out to be a foreign key referencing a column that wasn't unique. The error message pointed at the child table, but the fault was on the parent. The lesson: read the constraint definition, not the column it complains about.

Also discovered that ON DELETE CASCADE propagates through every layer of foreign key. One deletion at the top wiped four tables. This is either a feature or a footgun depending on whether you meant it.

Started the library ER diagram thinking it would take an hour. Took most of the weekend, because every relationship raised a question about how the system actually behaves. Can a member borrow the same copy twice? Is a reservation an entity or an attribute on a copy?

Practice

Mock interviews

Database questions of the kind asked in real interviews. Each one written out the way you'd actually answer at a whiteboard.

Q01 What's the difference between a primary key and a unique constraint? +

A primary key uniquely identifies each row and implies NOT NULL. A unique constraint enforces uniqueness but permits one or more nulls, depending on the engine.

A table has at most one primary key but can have many unique constraints. Foreign keys conventionally reference the primary key, but most engines let them reference any unique column too.

Q02 Explain the difference between INNER, LEFT, RIGHT, and FULL OUTER JOIN. +

INNER JOIN returns rows where the join condition matches on both sides. LEFT JOIN returns all rows from the left table and matching rows from the right (nulls where there's no match). RIGHT JOIN is the mirror image. FULL OUTER JOIN returns everything from both sides, with nulls where there's no match in either direction.

In practice, LEFT JOIN is the workhorse. RIGHT JOIN can almost always be rewritten as a LEFT JOIN with the tables swapped, which is easier to read.

Q03 What is a transaction, and what does ACID guarantee? +

A transaction is a unit of work the database treats as indivisible. ACID is the shorthand for four guarantees: Atomicity (all or nothing), Consistency (constraints hold before and after), Isolation (concurrent transactions don't see intermediate state), and Durability (committed work survives a crash).

Isolation is the only one that's tunable. Stricter isolation means less concurrency, so most engines default to Read Committed and let you opt up.

Q04 When would you choose NoSQL over a relational database? +

When the shape of the data, the access pattern, or the scale doesn't fit cleanly into rows and joins. Some honest examples: a key-value store as a cache where the only operation is "get by id." A document store when the natural unit of read and write is a deeply nested object. A graph store when the queries are mostly about traversing relationships several hops deep.

"NoSQL is faster" is usually the wrong answer. The right answer is that the data model fits the problem better.

Q05 What does an index actually do, and what's the cost? +

An index is a separate data structure (usually a B-tree) the database maintains alongside the table. It lets the engine find rows matching a predicate without scanning the whole table.

The cost is twofold. Indexes take space, and every insert, update, or delete on the table has to update the index too. So indexes speed up reads and slow down writes. The art is picking which columns are worth that tradeoff.

Q06 Explain normalization. What's 3NF and when would you violate it? +

Normalization is the process of removing redundancy and update anomalies from a schema. Each normal form is stricter than the last. Third Normal Form requires every non-key column to depend on the key, the whole key, and nothing but the key.

You'd violate 3NF deliberately when reads are heavy enough that joins become a bottleneck and you've measured the cost. A denormalized analytics table, a precomputed aggregate, a flat read model on top of an event log. Always intentional, never by accident.
