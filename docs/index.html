02

The relational model

Codd's 1970 paper is the foundation everything else is measured against. Data lives in relations (tables), each row a tuple, each column an attribute drawn from a domain. Constraints keep it honest.

Relation
A set of tuples sharing the same attribute structure. A table, in practice.
Tuple
A single row. An assertion that a fact holds.
Attribute
A named column drawn from a domain (its type).
Primary Key
The minimal set of attributes that uniquely identify each tuple.
Foreign Key
A column referencing the primary key of another relation. The glue.
Constraint
A rule the data must satisfy. Enforced by the database, not the application.
Worth remembering

Relational doesn't mean "tables with relationships between them." It means "every value is addressable by table, key, and attribute." That property is what makes set operations work.

03

SQL reference

SQL has more dialects than spoken English. The fundamentals carry across all of them, but the moment you reach for window functions, JSON operators, or anything date-related, check the docs for your engine.

SQL -- Find the top three customers by lifetime revenue, -- only counting paid orders from the last two years. SELECT c.id, c.name, SUM(o.total) AS lifetime_revenue, COUNT(*) AS order_count FROM customers c JOIN orders o ON o.customer_id = c.id WHERE o.status = 'paid' AND o.placed_at > NOW() - INTERVAL '2 years' GROUP BY c.id, c.name ORDER BY lifetime_revenue DESC LIMIT 3;

Clauses in the order they actually run

The order you write SQL in is not the order the engine evaluates it. Knowing the real order makes debugging faster and explains why you can't reference a SELECT alias in a WHERE clause.

Eval order 1. FROM // pick the source tables 2. JOIN // combine them 3. WHERE // filter rows 4. GROUP BY // fold into groups 5. HAVING // filter groups 6. SELECT // project columns 7. DISTINCT // dedupe 8. ORDER BY // sort 9. LIMIT // trim
04

Entity-relationship diagrams

ER diagrams are how a schema gets argued about before anyone writes CREATE TABLE. Entities are nouns. Relationships are verbs. Cardinality answers how many.

1 — 1

One-to-one

A user has one profile; a profile belongs to one user. Often a sign you could collapse them into one table unless you have a reason not to.

1 — N

One-to-many

A customer has many orders; an order belongs to one customer. The foreign key lives on the many side. The bread and butter of schema design.

N — N

Many-to-many

Students take many courses; courses have many students. Requires a join table holding both keys. Carry your join tables proudly.

05

Normalization

Normalization is the process of removing redundancy and update anomalies from a schema. Each normal form is stricter than the last. In practice most production schemas live around 3NF and selectively denormalize for read performance.

1NF
First Normal Form
Every column holds atomic values. No comma-separated lists, no nested structures.
Atomic
2NF
Second Normal Form
1NF plus every non-key column depends on the whole primary key, not part of it.
No partial
3NF
Third Normal Form
2NF plus no transitive dependencies. Non-key columns depend only on the key.
No transitive
BCNF
Boyce-Codd Normal Form
A stricter version of 3NF. Every functional dependency has a superkey on the left.
Stricter
Heuristic

Normalize until it hurts. Denormalize until it works. The first part is the design exercise. The second part is the optimization exercise. Don't skip straight to the second.

06

Indexing

An index is a separate data structure the database maintains to answer certain queries fast. The most common is a B-tree. Indexes speed up reads and slow down writes. Picking which columns to index is half the job of a database engineer.

B-tree
Default for most engines. Balanced tree, good for equality and range queries.
Hash
Equality only. Faster than B-tree for that one job, useless for ranges.
Composite
Multiple columns in one index. Order matters — leftmost prefix rules apply.
Covering
An index that contains every column the query needs. No table lookup required.
Partial
An index only over rows matching a predicate. Smaller, faster, fitter.
07

Transactions & ACID

A transaction is a unit of work the database treats as indivisible. ACID is the shorthand for the guarantees a database makes about that work.

A

Atomicity

All operations succeed, or none of them do. No half-finished work survives a crash.

C

Consistency

Every transaction leaves the database in a valid state. Constraints hold before and after.

I

Isolation

Concurrent transactions don't see each other's intermediate state. The level is configurable.

D

Durability

Once committed, the change survives a power loss, a crash, a restart. The disk remembers.

Isolation levels

Stricter isolation means less concurrency. Each level prevents certain anomalies and permits others. Most engines default to Read Committed.

Levels READ UNCOMMITTED // permits dirty reads READ COMMITTED // permits non-repeatable reads, phantom reads REPEATABLE READ // permits phantom reads SERIALIZABLE // permits nothing. As if run one at a time.
08

The NoSQL landscape

NoSQL is not one thing. It's a loose family of databases that gave up some of the relational model's strictness to gain something else — usually scale, flexibility, or a closer match to a specific access pattern.

Document

MongoDB, Couchbase

Stores nested documents (typically JSON). Good when the shape of your data is the unit you read and write.

Key-Value

Redis, DynamoDB

The simplest possible model. Lookup by key, get value. Fast, narrow, often a cache.

Wide-Column

Cassandra, ScyllaDB

Rows have flexible column sets. Designed for writes at scale across distributed clusters.

Graph

Neo4j, ArangoDB

First-class edges between nodes. The right tool when relationships are the query.

CAP theorem

In a distributed system you get to pick two of consistency, availability, and partition tolerance. Partitions happen whether you want them to or not, so the real choice is between consistency and availability when the network splits.
